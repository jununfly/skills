# REFERENCE — 五类手术关注点

每类手术都遵循 SKILL.md 的 8 步主流程，下面只补充类型特定的关注点。

---

## 1. Features 增删改（最大化复用现有代码库）

**核心心法**：宁可改 3 处旧代码，不要新建 1 个并行实现。

### Step 2 · 目标确认 特别关注
- 目标态：feature list 变成什么样（增/删/改哪些条目）

### Step 4 · 影响盘点 特别关注
- 列出当前 features list（按 module/route/command/menu 等用户可见维度）
- 对每个新需求，先问：现有哪个 feature 最接近？能否扩展参数/配置/分支？
- 标记"近似实现" → 优先复用；"无相关实现" → 才允许新建

### Step 6 · 编排路线 特别关注
- 增：先改宿主模块（挂新分支/参数），再改注册入口
- 删：先摘入口和注册点 → 再删实现 → 最后清依赖，反序安全
- 改：保持外部签名稳定，内部演进；签名必须改时走类型 3

### 反模式
- 复制粘贴一份相似模块然后小改 → 永远不要
- 新建一套并行的 utils/helpers → 必须先证明现有的不能扩展

---

## 2. 模式批量替换（coding-style / API 风格）

**核心心法**：先试点 → 再批量 → 最后兜底。

### Step 4 · 影响盘点 特别关注
- 用 Grep 精确匹配模式，统计命中数与文件分布
- 抽样 5-10 处审查：是否所有命中都该改？有没有特殊上下文要排除？
- 输出三档清单：**白名单**（确定改）+ **黑名单**（确定不改）+ **待定**（需用户决策）

### Step 6 · 编排路线 特别关注
- 先在 1-2 个文件试点 → 确认风格和测试通过 → 用户确认后扩展全量
- 批量改写时按目录或文件分批，每批可独立验证
- "待定"档逐个人工决策，不批量

### 工具选择
- 简单字面替换：编辑工具的 `replace_all`
- 涉及结构（参数顺序、签名、import 路径）：模型逐个理解后改写，不要硬塞 sed/awk
- 改完跑：格式化器 + linter + 类型检查 + 单测

---

## 3. 精准模块改写（保上下游不断裂）

**核心心法**：先画依赖图，再下刀。

### Step 4 · 影响盘点 特别关注
- 列出目标 subsystem/module/component/function 的：
  - 调用方（who calls me）
  - 被调方（who I call）
  - 类型/接口契约
  - 相关测试与 mock
  - 文档/注释中提及它的位置

### Step 6 · 编排路线 特别关注
- 先冻结外部契约（签名、返回值、错误类型），内部任意改
- 必须改契约时：①新增兼容版本 ②迁移所有调用方 ③删除旧版本，分三次提交
- 每改一处调用方，立即跑该处相关测试

### 红线
- 不要"顺手"改无关的相邻函数
- 不要扩大模块边界（除非用户明确要求）
- 不要在同一次改写里同时改"行为"和"结构"——分开做

---

## 4. 重构机会发现

**核心心法**：报告机会，不动刀。

### Step 2 · 目标确认 特别关注
- 用户想看哪个方面的重构机会？性能？可读性？可测性？解耦？

### Step 5 · 手术方案 输出格式
对每个发现项给出：
- **位置**：文件 + 行号
- **现状**：当前是什么样
- **机会**：可以重构成什么
- **价值**：为什么值得（可读性/性能/可测性/解耦）
- **成本**：影响面 + 风险等级
- **建议优先级**：P0 立即 / P1 近期 / P2 有空再做

### 常见信号
- 多处重复结构（DRY 候选）
- 长参数列表 / 隐式参数
- 上帝类、上帝函数
- 跨层调用、循环依赖
- 抽象漏洞（接口与实现耦合）
- 测试难以编写的代码（多半设计有问题）

### 红线
- 不要在诊断报告里塞"我已经顺手改了"——发现 ≠ 执行
- 不要给出超过 20 个发现项；多了就分主题
- 不要建议"重写整个模块"——那不是手术刀

---

## 5. 坏味道排查

**核心心法**：用证据说话，给可执行建议。

### Step 5 · 手术方案 输出格式
表格形式：味道类型 / 位置 / 证据片段 / 影响 / 建议手术类型（对应类型 1-3）

### 检查清单
- **重复代码**：相似函数、相似分支、复制粘贴痕迹
- **过长函数 / 过大文件**：>50 行函数、>500 行文件值得审视
- **过深嵌套**：>3 层 if/for 嵌套
- **魔法数 / 硬编码**：未抽常量的字面量
- **命名问题**：缩写不一致、误导性命名、单字母变量
- **死代码**：unused exports、unreachable branches、被注释的代码块
- **隐式依赖**：全局状态、单例滥用、隐藏的初始化顺序
- **错误处理**：吞异常、笼统 catch、缺乏边界检查
- **测试缺位**：核心路径无测试、测试本身坏味道

### 红线
- 不要把"风格偏好"包装成"坏味道"——区分客观问题与主观品味
- 给出位置必须可定位（文件 + 行号或函数名）
- 不要列超过 30 项；按严重度排序，超出的归入"其他"
