# Surgical Codebase Rewrite

> 门诊手术，不是大手术。先诊断，再方案，最后下刀；范围最小，溯源清楚。

## 核心原则

1. **方案先行**：未与用户对齐方案前，禁止动刀。
2. **最小侵入**：只改必须改的，不顺手"优化"无关代码。
3. **复用优先**：先找现有可复用的实现，再考虑新增。
4. **上下游闭环**：每一处改动都要追踪调用方/被调方/类型/测试。
5. **可回滚**：每次手术前用 git 留可回退的状态。

## Quick start

```
用户: 把 src/api/ 下所有 callback 风格改成 async/await，但别动 src/legacy/

你: 1. git stash / commit 保存现状
    2. 确认目标：callback → async/await，范围 src/api/，排除 src/legacy/
    3. 澄清边界：哪些子目录？测试文件改不改？
    4. 盘点影响：列出所有 callback 函数 + 调用方 + 相关测试
    5. 输出手术方案 → 等用户确认
    6. 编排执行路线：试点 → 批量 → 兜底
    7. 按切片逐个改写，每片验证
    8. 闭环验证：测试/lint/类型检查
```

## 8 步流程

### Step 1 · 现状保存
- [ ] git commit 或 git stash，确保可回滚
- [ ] 记录当前测试/lint 基线（通过/失败状态）

### Step 2 · 目标确认
- [ ] 和用户对齐：要达到什么状态？改完之后长什么样？
- [ ] 用一句话描述目标态，让用户确认

### Step 3 · 边界澄清
- [ ] 明确范围：目录/文件/模块，拒绝"全代码库"这种发散指令
- [ ] 明确排除项：不动的目录/文件/模式
- [ ] 明确验收标准：怎么算"改完了"

### Step 4 · 影响盘点
- [ ] 列出待改文件清单
- [ ] 追踪上下游依赖（调用方/被调方/类型/测试）
- [ ] 标记风险点：跨模块影响、签名变更、隐式依赖

### Step 5 · 手术方案
- [ ] 写成 checklist，逐项可勾可验
- [ ] 给用户审一次，确认后才进入执行
- [ ] checklist 超过 10 项 → 强制拆分为多次手术

### Step 6 · 编排路线
- [ ] 确定执行顺序：先低风险后高风险、先叶子后根、先试点后批量
- [ ] 识别可并行的独立切片
- [ ] 标注每个切片的前置依赖

### Step 7 · 切片执行
- [ ] 一次只动一个切片
- [ ] 改完立刻验证（语法/类型/单测）
- [ ] 发现超出方案的新问题 → 暂停，记入 followup，不自行扩大范围

### Step 8 · 闭环验证
- [ ] 跑全量相关测试 + lint + 类型检查
- [ ] 影响面回归：关键调用方行为是否一致
- [ ] 输出手术报告

## 五类手术

详见 [REFERENCE.md](REFERENCE.md)。简表：

| 类型 | 关键动作 | 必要前置 |
|---|---|---|
| 1. Features 增删改 | 在现有结构上挂/摘 feature | 找出可复用的脚手架 |
| 2. 模式批量替换 | 统一 coding-style / API 风格 | 先在 1-2 处试点 |
| 3. 精准模块改写 | 改函数签名/模块边界，保上下游不断 | 列全调用方 |
| 4. 重构机会发现 | 识别可抽象/可合并/可下沉的结构 | — |
| 5. 坏味道排查 | 长函数、重复代码、循环依赖、隐式耦合等 | — |

## 红线

- ❌ 用户没确认方案就动刀
- ❌ 把"顺手优化"塞进当前手术（记进 followup 列表）
- ❌ 改动超出用户声明的范围
- ❌ 删除测试或注释来"让它通过"
- ❌ 把"全代码库扫描"当成有效指令——必须收敛到具体范围

## Followup

每次手术结束，输出：
- ✅ 已完成项
- ⚠️ 发现但未处理的问题（留给下次门诊）
- 🧪 验证结果（测试/lint/类型检查）
